# Review: Principal Engineer
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The requirements artifacts cover the core projection loop -- income, expenses, contributions, withdrawals, taxes, and account balances -- with clean path references to the conceptual model. However, the requirements as a whole suffer from three systemic problems: every table is missing the mandatory Acceptance Criteria column prescribed by [requirements/principles.md](../requirements/principles.md); the denomination abbreviations in the conceptual model contradict the bedrock principles; and `conceptual-model.md` does not exist on disk, breaking all 92 relative links across 12 files. Beyond those systemic issues, several cross-cutting gaps -- plan termination, deficit-to-withdrawal linkage, early SS claiming, and MAGI computation -- leave the projection loop underspecified for design.

## Findings

| <a id="PE-no-ac-CRT-01"></a>PE-no-ac-CRT-01 | All engine and UX requirements files (e.g., [denomination.md](../requirements/engine/denomination.md), [assets.md](../requirements/engine/assets.md), [cli.md](../requirements/ux/cli.md)) | Every requirements table uses three columns (`Path | Tag | Requirement`) but [requirements/principles.md](../requirements/principles.md) mandates four columns: "Path, Tag, Requirement, Acceptance Criteria." | <a id="PE-broken-links-CRT-03"></a>PE-broken-links-CRT-03 | All engine and UX requirements files referencing `../conceptual-model.md` | `conceptual-model.md` does not exist on disk at `requirements/conceptual-model.md`. The file appears to be delivered only via the `@requirements/conceptual-model.md` directive in CLAUDE.md, which works for AI context assembly but not for human navigation or any tool that resolves markdown links. | Create `requirements/conceptual-model.md` on disk so that all relative links resolve correctly for both human and automated consumers. |
| <a id="PE-no-plan-end-MAJ-01"></a>PE-no-plan-end-MAJ-01 | [timeline.md](../requirements/engine/timeline.md) | The conceptual model states each member's stream spans "birth year to death year," and `D:streams / timeline` says "The plan timeline spans from the earliest stream start to the latest stream end." However, no requirement defines a configurable life expectancy, death year, or plan termination condition for any member. Without this, the projection range is undefined -- design cannot determine when the timeline ends. | Add a requirement for a configurable plan end condition per member (e.g., assumed death age or life expectancy) under `R:engine / timeline / [ member ]`. |
| <a id="PE-deficit-flow-MAJ-02"></a>PE-deficit-flow-MAJ-02 | [projection.md](../requirements/engine/projection.md) and [withdrawals.md](../requirements/engine/withdrawals.md) | `R:engine / projection / surplus-deficit` requires computing "total income after tax minus total expenses." `R:engine / withdrawals : sequence` defines a withdrawal priority sequence. But no requirement connects these two concepts -- there is no statement that a deficit triggers withdrawals from accounts per the sequence, or that a surplus flows into deposits. The projection loop is incomplete: design cannot determine what happens after a deficit is computed. | Add requirements that (a) a deficit triggers withdrawals from accounts per the configured withdrawal sequence, and (b) a surplus is deposited per a configurable deposit priority (or into the bank account by default). |
| <a id="PE-early-ss-MAJ-03"></a>PE-early-ss-MAJ-03 | [income.md](../requirements/engine/income.md) and [assumptions.md](../requirements/engine/assumptions.md) | `R:engine / income / ss / [ member ] : claiming-adjustment` requires adjusting SS benefits "based on claiming age relative to FRA." `R:engine / assumptions / policy / ss / delayed-credit-rate` specifies the increase rate for claiming after FRA. Without this, the claiming adjustment requirement is only half-specified. | Add a requirement for the early claiming reduction rate (e.g., `R:engine / assumptions / policy / ss / early-reduction-rate`) and ensure the claiming-adjustment requirement references both early and delayed adjustments. |
| <a id="PE-ss-range-MAJ-04"></a>PE-ss-range-MAJ-04 | [timeline.md](../requirements/engine/timeline.md), line 10 | `R:engine / timeline / [ member ] : ss-claiming-age / range-constraint` states: "The Social Security claiming age must fall within the range defined by `S:assumptions / policy / ss / fra`." But FRA is a single age (derived from birth year), not a range. The valid SS claiming range is 62 to 70, neither of which is defined in the conceptual model or the assumptions requirements. The constraint as written is unimplementable. | Define the SS claiming age range (minimum 62, maximum 70) as a policy assumption (e.g., `S:assumptions / policy / ss / claiming-range`) and update the range-constraint requirement to reference it instead of FRA. |
| <a id="PE-magi-MAJ-05"></a>PE-magi-MAJ-05 | [assumptions.md](../requirements/engine/assumptions.md), line 14 | `R:engine / assumptions / policy / limits / roth-ira / phase-out` requires: "Roth IRA direct contributions must phase out based on MAGI." MAGI (Modified Adjusted Gross Income) is referenced as an input but no requirement defines how MAGI is computed. MAGI differs from taxable income and AGI. Without a computation requirement, the phase-out cannot be evaluated. | Add a requirement for MAGI computation (e.g., `R:engine / projection / magi`) that specifies which income sources contribute to MAGI. |
| <a id="PE-income-earning-MAJ-06"></a>PE-income-earning-MAJ-06 | [timeline.md](../requirements/engine/timeline.md), [contributions.md](../requirements/engine/contributions.md), [income.md](../requirements/engine/income.md), [assumptions.md](../requirements/engine/assumptions.md) | The term "income-earning" is used as a qualifier on `S:household / [ member ]` in six requirements (e.g., "Each income-earning member must have a configurable retirement year") but is never defined. It implies a classification of household members, but the conceptual model and household requirements have no such attribute. A dependent could earn income; a spouse might not. | <a id="PE-two-member-tax-QST-01"></a>PE-two-member-tax-QST-01 | [projection.md](../requirements/engine/projection.md), [household.md](../requirements/engine/household.md) | The household requirements support a spousal relationship (`R:engine / household / [ member ] / relations / spouse`), and the conceptual model defines tax brackets by filing status (MFJ, single). ## Passed Checks

- **Path syntax consistency**: All requirements use the correct `R:` prefix path syntax with `/` composition and `[ ]` set notation, consistent with [requirements/principles.md](../requirements/principles.md).
- **Tag usage**: CORE, MVP, and FUT tags are used appropriately. CORE is reserved for foundational invariants (denomination, timeline, balance floor). MVP covers the projection feature set. FUT is used sparingly for alternative recurrence formulas and withdrawal sequences.
- **No solutioning**: Requirements are stated in terms of behaviors and constraints, not implementations. No requirements prescribe data structures, algorithms, file formats, or UI components. The "configurable match rate applied to deferrals up to a configurable deferral ceiling" in contributions is borderline but acceptable as a domain description.
- **One behavior per requirement**: Each row in the requirements tables addresses a single behavior. No compound requirements were observed.
- **Domain language**: Requirements use financial planning vocabulary (contributions, deferrals, RMDs, COLA, FRA, MAGI) rather than technical vocabulary. The exception is "JSON-formatted output" in the CLI requirements, which is acceptable as a testable output format constraint.
- **Conceptual model coverage**: The conceptual model's stream hierarchy is well-structured and comprehensive. The S: and D: path system provides clear traceability from requirements to model entities.
- **Cross-file reference consistency**: Where requirements reference other requirements files (e.g., expenses referencing timeline for retirement-year phase boundary), the references use correct relative links.
- **Bedrock principle alignment**: The stream-based computation model is consistently reflected in the requirements. Time-varying quantities (inflation rates, contribution limits, tax brackets) are modeled as streams with carry-forward semantics per bedrock principles.

## Out of Scope

- **Non-MVP placeholder files**: Eight UX files and three engine files (persistence, validation, stress-testing) are marked as not part of MVP. These were read but not reviewed for content since they contain only placeholder descriptions.
- **Per-formula correctness**: Individual formula accuracy (balance recurrence, RMD divisor application, tax bracket computation) is a Leaf concern.
- **Per-field completeness**: Whether every attribute in the conceptual model has a corresponding requirement is a Leaf audit, not a Forest concern. The gaps flagged above are cross-cutting (they affect multiple subsystems or break the projection loop).
- **UX layout and interaction design**: How the CLI table is formatted, column ordering, and display preferences are design concerns.
