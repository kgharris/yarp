# Review: Product Owner
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The MVP requirements cover the core projection engine well: household, accounts, contributions, withdrawals, income, expenses, tax, and a CLI output. The stream-based conceptual model is strong and internally consistent. However, the requirements are systematically missing acceptance criteria (a violation of [requirements principles](../requirements/principles.md)), several end-to-end user workflows are absent, and there are meaningful gaps between what the conceptual model promises and what the MVP specification actually covers -- particularly around account types that appear in MVP-tagged requirements but lack contribution/withdrawal/projection coverage.

## Findings

| ID | Location | Issue | Recommendation |
|---|---|---|---|
| <a id="PO-no-ac-CRT-01"></a>PO-no-ac-CRT-01 | All engine and UX requirements files | **Every requirement is missing acceptance criteria.** The requirements principles state: "A requirement without acceptance criteria is a wish. Every requirement must include conditions under which it can be declared satisfied." The specification table format defined in [principles.md](../requirements/principles.md) has four columns: Path, Tag, Requirement, and Acceptance Criteria. Every requirements file uses only three columns (Path, Tag, Requirement) and omits the Acceptance Criteria column entirely. This means no requirement in the MVP can be declared satisfied by an external observer. | Add the Acceptance Criteria column to every requirements table and populate it for each row. Criteria should describe observable conditions, not internal state. |
| <a id="PO-brokerage-gap-CRT-02"></a>PO-brokerage-gap-CRT-02 | [assets.md](../requirements/engine/assets.md), [contributions.md](../requirements/engine/contributions.md), [withdrawals.md](../requirements/engine/withdrawals.md), [projection.md](../requirements/engine/projection.md) | **Brokerage account exists in conceptual model and MVP candidates but has no MVP requirements.** The conceptual model defines `S:assets / accounts / [ member ] / [ brokerage ]` and `S:assets / accounts / [ brokerage ]`. The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) explicitly include "Track taxable brokerage: post-tax deposits, withdrawals (simplified tax treatment)." However, no MVP-tagged requirement in [assets.md](../requirements/engine/assets.md), [contributions.md](../requirements/engine/contributions.md), or [withdrawals.md](../requirements/engine/withdrawals.md) addresses brokerage accounts. The bank account is the only non-retirement account with MVP requirements. A user with a brokerage account -- common for retirement planners -- cannot model it. | Either add MVP requirements for brokerage accounts (tax treatment, contributions, withdrawals) or explicitly document the exclusion. The MVP candidates document suggests it was intended to be included. |
| <a id="PO-ira-gap-CRT-03"></a>PO-ira-gap-CRT-03 | [assets.md](../requirements/engine/assets.md), [withdrawals.md](../requirements/engine/withdrawals.md) | **Traditional IRA has no asset requirements and incomplete withdrawal coverage.** The conceptual model defines `S:assets / accounts / [ member ] / [ ira ]` and [assumptions.md](../requirements/engine/assumptions.md) requires IRA contribution limits and deductibility phase-outs. But [assets.md](../requirements/engine/assets.md) has no row for traditional IRA tax treatment (contributions pre-tax/deductible, growth tax-deferred, withdrawals taxable). [withdrawals.md](../requirements/engine/withdrawals.md) has no IRA withdrawal or IRA RMD requirement, even though [assumptions.md](../requirements/engine/assumptions.md) defines RMD tables that would apply to IRAs. A user with a traditional IRA -- one of the most common retirement accounts -- cannot model its full lifecycle. | Add MVP requirements for traditional IRA in assets.md (tax treatment), contributions.md (if not already implied by the IRA limit requirement), and withdrawals.md (standard withdrawal and RMD). |
| <a id="PO-no-first-run-MAJ-01"></a>PO-no-first-run-MAJ-01 | All requirements files | **No first-run or plan creation workflow is specified.** The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) include "Plan creation flow with pre-populated defaults" and "Shipped default configuration bundled; File-New creates functional plan." The assumptions requirements state every assumption "must have a shipped default value sufficient to produce a complete projection without user configuration." But there is no requirement describing the minimum data a user must provide to get their first projection. What does the user do when they first launch the tool? What inputs are required vs. defaulted? Without this, the tool has no defined entry point. | Add a requirement (likely in [cli.md](../requirements/ux/cli.md) or a new plan-creation file) describing the minimum user inputs needed to produce a first projection, and that all other values use shipped defaults. |
| <a id="PO-no-input-workflow-MAJ-02"></a>PO-no-input-workflow-MAJ-02 | [cli.md](../requirements/ux/cli.md) | **No requirement for how the user provides plan inputs.** The CLI file specifies output (projection table, JSON format) but says nothing about input. How does the user configure their household, accounts, contributions, income, expenses, and assumptions? Is it a configuration file? Command-line arguments? Interactive prompts? The MVP candidates mention "configuration screens" and "monetary inputs displayed in CNV, stored in YZV," but these did not make it into the actual requirements. The engine requirements define what must be configurable, but no UX requirement defines how the user configures it. | Add MVP requirements for the input mechanism. For a CLI MVP, this likely means a plan configuration file with a defined structure, or a set of CLI subcommands for plan setup. |
| <a id="PO-surplus-deficit-unresolved-MAJ-03"></a>PO-surplus-deficit-unresolved-MAJ-03 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / surplus-deficit` | **Surplus/deficit is computed but nothing says what happens with it.** The requirement states: "Each year must produce an observable surplus or deficit equal to total income after tax minus total expenses." But there is no requirement for what happens when there is a surplus (does it flow into an account? which one? the bank account?) or what happens when there is a deficit (does it trigger withdrawals from the withdrawal sequence?). The projection will produce a number, but the number has no behavioral consequence. This is the central cash-flow loop of the retirement plan and it is incomplete. | Add requirements that specify: (1) surplus is deposited into a designated account (e.g., bank), (2) deficit triggers withdrawals per the configured withdrawal priority sequence. This connects [projection.md](../requirements/engine/projection.md) to [withdrawals.md](../requirements/engine/withdrawals.md) and [contributions.md](../requirements/engine/contributions.md). |
| <a id="PO-no-net-worth-MAJ-04"></a>PO-no-net-worth-MAJ-04 | [projection.md](../requirements/engine/projection.md), [cli.md](../requirements/ux/cli.md) | **Net worth is not defined in engine requirements but is expected in output.** The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) include "Calculate net worth" and "Table showing: account balances, net worth, retirement pool." The CLI requirement `R:ux / cli / projection-table : columns` does not mention net worth, and no engine requirement defines how to compute it. For a retirement planner, net worth is a primary orientation metric. | Add an engine requirement for net worth computation (sum of all account balances for MVP, since liabilities and hard assets are excluded) and include it in the CLI projection table columns. |
| <a id="PO-roth-401k-gap-MAJ-05"></a>PO-roth-401k-gap-MAJ-05 | [assets.md](../requirements/engine/assets.md), [contributions.md](../requirements/engine/contributions.md), [withdrawals.md](../requirements/engine/withdrawals.md) | **Roth 401k appears in conceptual model but has zero MVP requirements.** `S:assets / accounts / [ member ] / [ roth-401k ]` is defined in the conceptual model with contribution streams (employee, catchup, employer, mega). No MVP requirement in assets, contributions, or withdrawals references Roth 401k. Many users have Roth 401k options through their employer. | Decide whether Roth 401k is MVP or FUT. If FUT, this is fine but should be documented in an exclusions section. If MVP, add requirements for its tax treatment, contributions, and withdrawals. |
| <a id="PO-hsa-gap-MAJ-06"></a>PO-hsa-gap-MAJ-06 | [assets.md](../requirements/engine/assets.md), [contributions.md](../requirements/engine/contributions.md), [expenses.md](../requirements/engine/expenses.md) | **HSA appears in conceptual model and expense requirements reference healthcare phases, but no HSA account requirements exist.** The conceptual model defines `S:assets / accounts / [ member ] / [ hsa ]` with contribution and withdrawal streams. The expenses requirements include employer health insurance, ACA, and Medicare phases -- suggesting the system understands health cost lifecycle. But there is no MVP requirement for the HSA account itself. | Decide whether HSA is MVP or FUT. The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) exclusion table lists HSA under "Advanced accounts" with reason "Complex eligibility rules." If excluded, this is consistent, but the gap between the detailed health expense phasing (which is MVP) and the absence of the HSA (also excluded) should be noted for users who fund health expenses from HSAs. |
| <a id="PO-member-death-year-MAJ-07"></a>PO-member-death-year-MAJ-07 | [timeline.md](../requirements/engine/timeline.md), [household.md](../requirements/engine/household.md) | **No requirement for member death year or plan end condition.** The conceptual model states each member has "a timeline stream spanning birth year to death year." The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) mention "configurable projection start year and end year (max span 100 years)." But no MVP requirement defines how the plan determines when to stop projecting. Is there a configurable death year? A fixed plan end year? Life expectancy assumption? Without this, the plan timeline defined in [D:streams / timeline](../requirements/conceptual-model.md#d-streams) has no definable end. | Add a requirement for plan end year or member life expectancy, and specify what happens to the projection when a member's timeline ends (especially for the surviving spouse in a two-member household). |
| <a id="PO-ss-claiming-range-MAJ-08"></a>PO-ss-claiming-range-MAJ-08 | [timeline.md](../requirements/engine/timeline.md): `R:engine / timeline / [ member ] : ss-claiming-age / range-constraint` | **SS claiming age constraint references FRA but does not specify the actual range.** The requirement says: "The Social Security claiming age must fall within the range defined by S:assumptions / policy / ss / fra." FRA is not a range -- it is a single age. The actual claimable range is 62 to 70. The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) correctly state "SS claiming age range 62-70." The requirement as written does not give the user or tester enough information to know what values are valid. | Rewrite to specify that the SS claiming age must be within the range 62 to 70 (inclusive), with benefit adjustments relative to FRA. |
| <a id="PO-bank-growth-MIN-01"></a>PO-bank-growth-MIN-01 | [assets.md](../requirements/engine/assets.md), [projection.md](../requirements/engine/projection.md) | **Bank account growth rate is unspecified.** `R:engine / assets / accounts / [ bank ]` defines the bank as "household cash reserve" and [projection.md](../requirements/engine/projection.md) requires all accounts to use the weighted-average allocation rate. But a bank account typically earns a different (often near-zero) rate. There is no requirement distinguishing bank account growth from investment account growth. If the bank account uses the same equity/bond blended rate, the projection will overstate cash reserves. | Add a requirement that the bank account growth rate is independently configurable (defaulting to 0% or a savings rate), separate from the allocation-weighted investment return. |
| <a id="PO-dependent-expenses-MIN-02"></a>PO-dependent-expenses-MIN-02 | [timeline.md](../requirements/engine/timeline.md), [expenses.md](../requirements/engine/expenses.md) | **Dependent age-out removes them from budget but no dependent-specific expenses exist.** `R:engine / timeline / [ member ] : dependent-age-out / budget-exit` says dependents "leave the household budget" at age-out. But [expenses.md](../requirements/engine/expenses.md) defines only household-level expenses (living, health insurance, out-of-pocket). There are no per-dependent expense streams in MVP. The conceptual model includes `S:expenses / [ member ] / education`, `S:expenses / [ member ] / childcare`, etc., but none are MVP-tagged. The age-out mechanism has no expenses to remove. | Either add at least one per-dependent expense stream (or a simple dependent cost adder) to make the age-out behavior meaningful, or defer the dependent-age-out requirement to FUT since it has no observable effect in the current MVP. |
| <a id="PO-catchup-contrib-MIN-03"></a>PO-catchup-contrib-MIN-03 | [contributions.md](../requirements/engine/contributions.md), [assumptions.md](../requirements/engine/assumptions.md) | **401k catchup and IRA catchup limits are defined in assumptions but have no contribution requirements.** [assumptions.md](../requirements/engine/assumptions.md) references `S:assumptions / policy / limits / 401k : catchup` and `S:assumptions / policy / limits / ira : catchup`, and the conceptual model defines `S:contributions / [ member ] / [ 401k ] / catchup` and `S:contributions / [ member ] / [ ira ] / catchup`. But [contributions.md](../requirements/engine/contributions.md) has no MVP requirement for catchup contributions. Users over 50 can contribute significantly more; this matters for retirement planning accuracy. | Add MVP requirements for age-based catchup contributions to 401k and IRA, or explicitly defer to FUT with a note that this affects projection accuracy for users over 50. |
| <a id="PO-allocation-per-account-QST-01"></a>PO-allocation-per-account-QST-01 | [assumptions.md](../requirements/engine/assumptions.md): `R:engine / assumptions / allocations` | **Is the asset allocation global or per-account?** The requirement says "the plan must support a configurable asset allocation" and the conceptual model defines per-account overrides (`S:assumptions / allocations / [ member ] / [ account ]`). The MVP candidates mention "pre-retirement and post-retirement allocation." The current requirement does not clarify whether MVP supports one global allocation, phase-based allocations, or per-account allocations. This is a significant usability question: most users have different allocations in their 401k vs. their IRA. | Clarify whether MVP supports a single global allocation, phase-based allocations (pre/post-retirement), per-account allocations, or some combination. A single global allocation is the simplest MVP. |
| <a id="PO-filing-status-QST-02"></a>PO-filing-status-QST-02 | [projection.md](../requirements/engine/projection.md), [assumptions.md](../requirements/engine/assumptions.md) | **Filing status is implied but never explicitly required.** Tax brackets are "by filing status" in the conceptual model (`S:assumptions / policy / tax / federal / brackets`). The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) mention "MFJ and single filing statuses." But no MVP requirement specifies that the plan must have a configurable filing status, or how it interacts with household composition (married = MFJ, single member = single). | Add a requirement for filing status as a configurable household attribute, with a default derivation rule based on household composition. |
| <a id="PO-income-growth-QST-03"></a>PO-income-growth-QST-03 | [income.md](../requirements/engine/income.md): `R:engine / income / w2 / [ member ] / [ employer ]` | **W-2 income growth/inflation is unspecified.** The requirement says income is "active through that member's retirement year" but does not say whether salary grows over time. For a multi-decade projection, a flat nominal salary produces very different results than one that grows with inflation or at a configured raise rate. The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) mention "configurable amount" but not growth. | Decide whether W-2 income inflates at CPI, grows at a configurable rate, or remains flat. Add a requirement specifying the behavior. A configurable annual raise rate (defaulting to CPI) is the most useful MVP approach. |
| <a id="PO-roth-conversion-QST-04"></a>PO-roth-conversion-QST-04 | [withdrawals.md](../requirements/engine/withdrawals.md) | **Roth conversions are in the conceptual model but not addressed in MVP.** `S:withdrawals / roth-conversion / [ member ]` is defined in the conceptual model. The [MVP candidates](../reviews/old/mvp-requirements-candidates.md) exclude "Roth conversion ladder" under "Optimization strategies." This is consistent, but Roth conversions are one of the most impactful retirement planning strategies. | No action needed for MVP if this is a deliberate deferral. Consider prioritizing for the first post-MVP release since the withdrawal and contribution infrastructure is already in place. |

## Passed Checks

- **Conceptual model alignment:** The engine requirements files map cleanly to the conceptual model tree structure. Path syntax is used correctly and consistently.
- **Tag usage:** CORE, MVP, and FUT tags are used appropriately. CORE is reserved for true invariants (denomination, timeline, account balance floor). MVP items are a reasonable minimal set.
- **One behavior per requirement:** Most requirements are well-scoped to a single testable behavior. No egregious compound requirements found.
- **No solutioning:** The requirements are technology-agnostic. No algorithm choices, data structures, or UI component specifications are embedded.
- **Domain language:** Requirements use retirement planning vocabulary throughout. A financial advisor could read and validate them.
- **Bedrock principle adherence:** Stream-based computation, denomination frames, and MVC separation are respected. No requirement violates the bedrock architecture.
- **Expense phasing:** Health insurance lifecycle (employer -> ACA -> Medicare) is well-defined with clear phase boundaries tied to member milestones.
- **Deferred content:** Files marked "NOT PART OF MVP" are appropriately placeholders. The exclusion boundaries are reasonable.
- **Withdrawal sequencing:** The default priority sequence (bank -> 401k -> Roth IRA) is sensible and the configurable override is properly deferred to FUT.
- **SS benefit modeling:** PIA-based computation with claiming age adjustments, FRA derivation, and COLA are all present and correctly structured.
- **Inflation architecture:** CPI, CPI-W, and health inflation are properly separated with sensible MVP defaults (health and CPI-W default to CPI).

## Out of Scope

- Formula-level correctness (e.g., whether the end-of-year recurrence relation is mathematically stated correctly) -- that is a Leaf/Branch concern.
- Per-field data type or precision requirements -- that is design/implementation.
- Comparison with the old requirements extraction (~2,400 requirements in the old format) -- this review assesses the current specifications on their own terms.
- Non-MVP placeholder files (persistence, validation, stress-testing, dashboard, projections-view, budget-view, balance-sheets, scenarios, estate, charts, data-management) -- intentionally deferred.
- Wording and formatting consistency within individual requirements -- minor editorial concerns below the Product Owner's altitude.
