# Review: UX QA Engineer
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The MVP UX surface is intentionally narrow -- a single CLI projection table with JSON output. This makes the review focused but not trivial: the six requirements in [cli.md](../requirements/ux/cli.md) are the entire user-visible interface, so gaps here are gaps the user will hit directly. The requirements cover the happy path (run projection, see table) but leave several failure-mode and edge-case user flows unspecified. Most findings concern what the user sees when something goes wrong, what they can do about it, and how they know they are looking at the right numbers.

## Findings

| ID | Location | Issue | Recommendation |
|---|---|---|---|
| <a id="UXQA-err-msg-CRT-01"></a>UXQA-err-msg-CRT-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : missing-assumptions` | The requirement states "The projection must not run if required assumptions are missing" but does not specify what the user sees when this happens. A silent exit, an opaque error code, or a stack trace all satisfy the current wording. The user needs to know *which* assumptions are missing and how to supply them. Without this, the user's only recourse is trial and error. | Add an acceptance criterion: the CLI must identify each missing assumption by name and indicate where to configure it. |
| <a id="UXQA-err-msg-CRT-02"></a>UXQA-err-msg-CRT-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : allocation-sum` | Same class of issue as UXQA-err-msg-CRT-01. "The projection must not run if allocations do not sum to 100%" says nothing about what the user sees. A user who enters allocations summing to 99% or 101% needs to see the actual sum and which allocations are involved. | Add an acceptance criterion: the CLI must report the actual sum and the individual allocation values so the user can identify the discrepancy. |
| <a id="UXQA-no-ac-CRT-03"></a>UXQA-no-ac-CRT-03 | [cli.md](../requirements/ux/cli.md): all rows | No requirement in this file has acceptance criteria. [Requirements principles](../requirements/principles.md) state "A requirement without acceptance criteria is a wish. Every requirement must include conditions under which it can be declared satisfied." The table has Path, Tag, and Requirement columns but no Acceptance Criteria column. Every requirement in this file lacks the fourth column mandated by the specification conventions. | Add an Acceptance Criteria column to every row. Each criterion should describe observable CLI output or behavior that confirms the requirement is met. |
| <a id="UXQA-denom-label-MAJ-01"></a>UXQA-denom-label-MAJ-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : denomination` | The requirement says monetary values must be displayed per `D:denomination / display`, which defaults to CYV (current-year value). However, there is no requirement that the output *label* which denomination is in use. A user looking at a table of dollar amounts has no way to know whether they are in current-year dollars, nominal dollars, or year-zero dollars -- especially if a future requirement adds denomination switching. Even in MVP, the user should know the basis of the numbers they are reading. | Add a requirement that the projection table output must indicate the active denomination reference frame (e.g., "All values in 2026 dollars"). |
| <a id="UXQA-json-err-MAJ-02"></a>UXQA-json-err-MAJ-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / output-format : json` | The JSON output requirement says nothing about error reporting in JSON mode. If the CLI produces errors (missing assumptions, invalid allocations), must those errors also be structured as JSON? Test infrastructure consuming JSON output will break or produce misleading results if errors arrive as unstructured stderr text. | Add a requirement specifying that in JSON output mode, validation errors and failure conditions must also be emitted as structured JSON. |
| <a id="UXQA-json-schema-MAJ-03"></a>UXQA-json-schema-MAJ-03 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / output-format : json` | The JSON output requirement says "The CLI must support JSON-formatted output for use by test infrastructure" but does not specify what constitutes a complete, correct JSON response. Without a requirement on what the JSON must contain (at minimum the same data as the table), an implementation could emit partial JSON or JSON with a different structure than the table and still satisfy the requirement. | Add a requirement that the JSON output must contain the same data set as the tabular projection output (all columns specified in `R:ux / cli / projection-table : columns`). |
| <a id="UXQA-empty-plan-MAJ-04"></a>UXQA-empty-plan-MAJ-04 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table` | There is no requirement governing what happens when the plan has zero accounts, zero income sources, or zero expenses. The engine requirements (e.g., [projection.md](../requirements/engine/projection.md) `R:engine / projection / account-balance`) assume accounts exist, but the CLI does not specify behavior for a minimal or empty plan. A user who runs the CLI before configuring any accounts should see a clear message, not an empty table or a crash. | Add a requirement specifying CLI behavior when the plan contains no accounts, no income, or no expenses -- at minimum, an informative message indicating the plan is incomplete. |
| <a id="UXQA-exit-code-MAJ-05"></a>UXQA-exit-code-MAJ-05 | [cli.md](../requirements/ux/cli.md): all rows | No requirement specifies CLI exit codes. For a command-line tool consumed by both humans and test infrastructure (per the JSON requirement), exit codes are the primary machine-readable signal for success vs failure. Without them, test scripts cannot reliably distinguish a successful projection from a validation failure. | Add a requirement that the CLI must exit with a distinct code for success and for each class of failure (missing assumptions, invalid allocations, etc.). |
| <a id="UXQA-col-order-MIN-01"></a>UXQA-col-order-MIN-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The requirement lists the columns (calendar year, ages, balances, income, expenses, federal tax, surplus/deficit) but does not specify their ordering or grouping. For a text-mode table, column order materially affects readability. Two implementations could order columns differently, producing visually inconsistent output across versions. | Add a requirement specifying column ordering, or at minimum state that columns must appear in the order listed. |
| <a id="UXQA-multi-member-MIN-02"></a>UXQA-multi-member-MIN-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The requirement says the table must include "ages of all household members" and "balances of all accounts." For a household with two adults, three dependents, and six accounts, this could produce an extremely wide table. There is no requirement addressing readability when the number of columns exceeds terminal width or when members/accounts need to be distinguishable. | Consider adding a requirement on how multi-member, multi-account households are presented -- whether columns are labeled by member/account name, whether accounts are grouped by member, and how wide output is handled. |
| <a id="UXQA-rmd-visibility-MIN-03"></a>UXQA-rmd-visibility-MIN-03 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The columns list includes "balances of all accounts" and "income by source" but does not mention RMD withdrawals. The engine computes and applies RMDs (per [withdrawals.md](../requirements/engine/withdrawals.md) `R:engine / withdrawals / [ member ] / [ 401k ] / rmd`), and they are a critical compliance item. RMDs affect both balances and taxable income, but the user cannot see the RMD amount itself in the current column list. A user verifying their plan against IRS requirements cannot confirm the RMD is correct without seeing it explicitly. | Consider adding RMD amounts (or a withdrawals-by-type breakdown) to the projection table columns, or note this as deferred to the projections-view. |
| <a id="UXQA-contrib-visibility-MIN-04"></a>UXQA-contrib-visibility-MIN-04 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | Similarly, contributions are not listed as a column. The engine enforces contribution limits (per [contributions.md](../requirements/engine/contributions.md)), and contributions are a major cash-flow item, but the user cannot see what was contributed to each account in each year. This limits the user's ability to verify the plan's behavior. | Consider adding contribution totals (or per-account contributions) to the column list, or note as deferred. |
| <a id="UXQA-surplus-sign-MIN-05"></a>UXQA-surplus-sign-MIN-05 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The "surplus or deficit" column is listed without specifying how surplus vs deficit is distinguished in the output. Is it a signed number (positive = surplus, negative = deficit)? Is there a label? In a text table, the convention for representing negative values (parentheses, minus sign, color) affects whether the user can quickly identify problem years. | Add a requirement or acceptance criterion specifying how surplus and deficit are visually distinguished in the CLI table output. |
| <a id="UXQA-input-spec-QST-01"></a>UXQA-input-spec-QST-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table` | The requirement says "The plan must produce a year-by-year projection table via command-line invocation" but does not specify how the plan data is provided to the CLI. Is it a file path argument? A directory? Standard input? Without this, there is no way to test the CLI end-to-end because the input mechanism is undefined. The engine [persistence.md](../requirements/engine/persistence.md) is marked non-MVP, so the source of plan data is not specified anywhere. | Clarify how the CLI receives plan data -- at minimum, specify that it reads from a file whose path is provided as a command-line argument. |
| <a id="UXQA-denom-select-QST-02"></a>UXQA-denom-select-QST-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : denomination` | The requirement references `D:denomination / display` which defaults to CYV. The bedrock principles describe denomination switching as a capability (CNV reference year can shift "based on user selection"). Is there an MVP mechanism for the user to request a different denomination on the CLI, or is CYV the only supported mode? If the latter, this should be stated explicitly. If the former, the switching mechanism needs a requirement. | Either add a requirement for a denomination flag/option on the CLI, or add an acceptance criterion stating that MVP output is always CYV with no switching. |
| <a id="UXQA-partial-val-QST-03"></a>UXQA-partial-val-QST-03 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : missing-assumptions` and `R:ux / cli / projection-table : allocation-sum` | These are the only two validation gates specified. But the engine has many other potential failure modes: a member with no birth year, an account with no initial balance, a retirement year that precedes the current year, a Social Security claiming age outside the valid range (per [timeline.md](../requirements/engine/timeline.md) `R:engine / timeline / [ member ] : ss-claiming-age / range-constraint`). Are all other validation failures out of scope for the CLI, or are these two simply the ones called out while a general "report all validation failures" requirement is expected? | Clarify whether the CLI must surface all engine validation failures or only the two listed. If only the two, state that other validation is deferred. |

## Passed Checks

- **Denomination reference:** The CLI requirement correctly references `D:denomination / display` from the conceptual model, maintaining proper cross-reference discipline.
- **Column coverage of engine outputs:** The columns listed (account balances, income by source, expenses, federal tax, surplus/deficit) align with the engine projection requirements (`R:engine / projection / account-balance`, `R:engine / projection / income-tax`, `R:engine / projection / surplus-deficit`).
- **Stream path references:** All `S:` and `D:` path references in [cli.md](../requirements/ux/cli.md) use valid conceptual model paths and include proper markdown links.
- **No solutioning:** The requirements are appropriately technology-agnostic. They specify what must appear (columns, denomination) without prescribing table formatting libraries, terminal APIs, or JSON serialization details.
- **Scope discipline:** The non-MVP files (dashboard, projections-view, budget-view, balance-sheets, scenarios, estate, charts, data-management) are properly marked as placeholders and do not contain premature requirements.

## Out of Scope

- **Engine calculation correctness:** Whether the balance recurrence formulas, RMD computations, or tax bracket applications are mathematically correct is an engine concern, not a UX QA finding.
- **Design-layer decisions:** Table formatting, JSON schema structure, terminal rendering approach, and similar design decisions are not assessed at the requirements level.
- **Non-MVP UX files:** The eight placeholder files (dashboard, projections-view, budget-view, balance-sheets, scenarios, estate, charts, data-management) contain no reviewable requirements. They will need full review when populated.
- **Persistence and data format:** How plan data is stored and loaded is an engine/design concern. The QST-01 finding notes the gap from the UX perspective but does not attempt to resolve it.
