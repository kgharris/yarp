# Review: Engine QA Engineer
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The engine requirements establish a clear stream-based architecture and cover the major financial domains (accounts, contributions, withdrawals, income, expenses, projections). However, nearly every requirement lacks acceptance criteria -- the requirements principles document mandates them ("A requirement without acceptance criteria is a wish") yet no artifact includes them. Beyond that systemic gap, there are critical testability concerns: no numerical precision or rounding requirements are stated anywhere, several computation rules have hidden ambiguities (e.g., the RMD computation lacks a balance-timing specification, the SS early-claiming reduction formula is absent), and the requirements contain no conservation-of-assets invariant despite the system moving money between accounts, income, expenses, and taxes every projection year.

## Findings

| <a id="EQA-no-ac-CRT-01"></a>EQA-no-ac-CRT-01 | All engine requirement files | **No acceptance criteria anywhere.** [requirements/principles.md](../requirements/principles.md) states "A requirement without acceptance criteria is a wish. Every requirement must include conditions under which it can be declared satisfied." The requirement tables have columns Path, Tag, and Requirement -- but no Acceptance Criteria column appears in any file. Every single requirement is therefore unverifiable against its own principles. | Add an Acceptance Criteria column to every requirement table and populate it with observable, testable conditions for each row. |
| <a id="EQA-no-precision-CRT-02"></a>EQA-no-precision-CRT-02 | All engine requirement files | **No numerical precision or rounding requirements.** The system performs multi-decade compound projections involving inflation, growth rates, tax brackets, and dollar amounts. No requirement specifies precision (e.g., "accurate to the cent"), rounding mode (e.g., round half-even, truncate), or acceptable cumulative tolerance. Without this, there is no way to write a test that distinguishes a correct answer from an approximately correct one. | Add a cross-cutting engine requirement specifying precision for dollar amounts, rounding rules, and acceptable tolerance over the projection horizon. |
| <a id="EQA-no-conservation-CRT-03"></a>EQA-no-conservation-CRT-03 | [projection.md](../requirements/engine/projection.md) | **No conservation-of-assets invariant.** Every dollar of income, contribution, withdrawal, expense, and tax should be accounted for. The requirements specify surplus/deficit (`R:engine / projection / surplus-deficit`) but do not require that the sum of all inflows minus all outflows equals the net change in total assets. Without this invariant, silent errors (double-counted contributions, missing tax withdrawals, leaked funds) cannot be detected. | Add a CORE requirement: for every projection year, net change in total account balances must equal total inflows minus total outflows minus taxes. |
| <a id="EQA-rmd-balance-timing-CRT-04"></a>EQA-rmd-balance-timing-CRT-04 | [withdrawals.md](../requirements/engine/withdrawals.md): `R:engine / withdrawals / [ member ] / [ 401k ] / rmd : computation` | **RMD computation does not specify which balance to use.** The requirement says to compute RMDs "using uniform-lifetime-table and start-age" but does not state whether the divisor is applied to the prior year-end balance (Dec 31) or some other point-in-time balance. IRS rules require the prior year-end balance. This is a domain-level behavioral requirement, not a design detail -- the RMD amount changes if the wrong balance is used, and a test cannot verify correctness without knowing which balance is expected. | Specify that RMD for year Y is computed from the account balance as of Dec 31 of year Y-1 divided by the applicable divisor. |

Without specifying this, early-claiming scenarios are untestable. | Add a requirement (or assumption stream) for the early-claiming reduction schedule and reference it from the claiming-adjustment requirement. |
| <a id="EQA-ss-claiming-range-MAJ-02"></a>EQA-ss-claiming-range-MAJ-02 | [timeline.md](../requirements/engine/timeline.md): `R:engine / timeline / [ member ] : ss-claiming-age / range-constraint` | **SS claiming age range is underspecified.** The requirement says the claiming age "must fall within the range defined by `S:assumptions / policy / ss / fra`." FRA is a single age, not a range. The actual SSA range is 62 to 70. This requirement as written does not communicate what the valid range is, making boundary testing impossible. | Specify the valid range explicitly (e.g., age 62 to age 70) or add an assumption stream for the SS claiming window. |
| <a id="EQA-401k-catchup-MAJ-03"></a>EQA-401k-catchup-MAJ-03 | [contributions.md](../requirements/engine/contributions.md) | **401k catchup and super-catchup contributions missing from requirements.** The conceptual model defines `S:contributions / [ member ] / [ 401k ] / catchup` and the assumptions define catchup and super-catchup limits in `S:assumptions / policy / limits / 401k`, but [contributions.md](../requirements/engine/contributions.md) contains no requirement for catchup contributions. Age-based catchup logic (50-59, 60-63 super-catchup, 64+) is untestable if not required. | Add MVP requirements for 401k catchup and super-catchup contributions, including the age-band boundaries (50-59/64+, 60-63). |

Members age 50+ are entitled to additional contributions. | Add an MVP requirement for Roth IRA catchup contributions referencing the IRA catchup limit and the age threshold. |
| <a id="EQA-hsa-missing-MAJ-05"></a>EQA-hsa-missing-MAJ-05 | [contributions.md](../requirements/engine/contributions.md), [withdrawals.md](../requirements/engine/withdrawals.md), [assets.md](../requirements/engine/assets.md) | **HSA accounts entirely absent from MVP requirements.** The conceptual model defines HSA accounts, contributions (employee and employer), and withdrawals (qualified and unqualified). | <a id="EQA-roth-401k-missing-MAJ-08"></a>EQA-roth-401k-missing-MAJ-08 | [assets.md](../requirements/engine/assets.md), [contributions.md](../requirements/engine/contributions.md), [withdrawals.md](../requirements/engine/withdrawals.md) | **Roth 401k entirely absent from MVP requirements.** The conceptual model defines Roth 401k accounts, contributions (employee, catchup, employer, mega), and withdrawals. | <a id="EQA-surplus-routing-MAJ-09"></a>EQA-surplus-routing-MAJ-09 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / surplus-deficit` | **Surplus/deficit has no routing requirement.** The requirement produces "an observable surplus or deficit equal to total income after tax minus total expenses" but does not specify what happens to it. A surplus presumably flows into accounts (bank? brokerage? according to what rule?) and a deficit presumably triggers withdrawals (per the withdrawal sequence?). Without specifying the routing, the projection loop cannot be verified end-to-end. | Add requirements specifying how surplus is deposited and how deficit triggers the withdrawal sequence. |
| <a id="EQA-tax-which-income-MAJ-10"></a>EQA-tax-which-income-MAJ-10 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / income-tax : taxable-income` | **Taxable income composition is not specified.** The requirement says "compute federal taxable income ... by applying standard-deduction" but does not enumerate which income sources are included in gross income. W-2 income, SS benefits (taxable portion), 401k withdrawals, and IRA withdrawals all have different inclusion rules. Without specifying the composition, a test cannot verify that taxable income is correct. | Add a requirement enumerating the income sources that compose gross income, their inclusion rules, and the order of deductions. |
| <a id="EQA-rate-derivation-allocation-MAJ-11"></a>EQA-rate-derivation-allocation-MAJ-11 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / account-balance : rate-derivation` | **Rate derivation does not address per-account allocation overrides.** The requirement states the effective rate is "the weighted average of rates using allocations as weights." But the conceptual model allows per-member-per-account allocation overrides (`S:assumptions / allocations / [ member ] / [ account ]`). The requirement does not specify whether a per-account override takes precedence, how the fallback chain works, or what happens when only some accounts have overrides. | Specify the allocation resolution order: per-account override, then household default, and confirm that the weighted-average formula applies at each resolution level. |
| <a id="EQA-floor-enforcement-MAJ-12"></a>EQA-floor-enforcement-MAJ-12 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / account-balance : floor` | **Account balance floor has no specified enforcement behavior.** "No account balance may go below zero" does not state what happens when a withdrawal or deficit would push a balance negative. Does the withdrawal get capped? Does the remainder cascade to the next account in the withdrawal sequence? Does the deficit carry forward? The enforcement mechanism determines correctness of every deficit-year test. | Specify the behavior when a withdrawal would reduce a balance below zero: cap the withdrawal at the available balance, and define how the shortfall is handled. |
| <a id="EQA-rmd-floor-interaction-MAJ-13"></a>EQA-rmd-floor-interaction-MAJ-13 | [withdrawals.md](../requirements/engine/withdrawals.md): `R:engine / withdrawals / [ member ] / [ 401k ] / rmd : floor` | **RMD floor interaction with voluntary withdrawals is ambiguous.** "The required minimum distribution is a floor on the total withdrawal" -- does this mean (a) if voluntary withdrawal >= RMD, no additional withdrawal is needed, or (b) the RMD is always withdrawn on top of voluntary withdrawals? Interpretation (a) is correct per IRS rules, but the wording "floor on the total withdrawal" could be read either way. A test with both voluntary and RMD withdrawals in the same year would produce different results. | Clarify: if the total voluntary withdrawal from the account in a given year meets or exceeds the RMD, no additional RMD withdrawal is required. | <a id="EQA-employer-ins-phase-MAJ-14"></a>EQA-employer-ins-phase-MAJ-14 | [expenses.md](../requirements/engine/expenses.md): `R:engine / expenses / health / insurance / employer : phase` | **Employer insurance phase boundary is ambiguous for two-member households.** The requirement says employer insurance is active "while at least one member has not reached retirement-year." If member A retires and member B keeps working, does the employer premium remain at the full household amount? Does it halve? This is a domain rule that affects the expense amount, not just the active/inactive state. | Clarify whether the employer insurance premium changes when one of two members retires, or whether the stream value remains constant as long as any member is working. |
| <a id="EQA-aca-phase-overlap-MAJ-15"></a>EQA-aca-phase-overlap-MAJ-15 | [expenses.md](../requirements/engine/expenses.md): `R:engine / expenses / health / insurance / aca : phase` and `R:engine / expenses / health / insurance / employer : phase` | **Health insurance phase transitions can overlap in two-member households.** If member A retires at 60 and member B retires at 62, with Medicare at 65, then from age 60-62 (A retired, B working) both employer and ACA could arguably be active. The employer phase says "active while at least one member has not reached retirement." The ACA phase says "active while at least one member has reached retirement but has not reached Medicare." In the 60-62 window, both conditions are true. This creates a year where both expense streams are active, which is likely incorrect. | Add a requirement that clarifies the mutual exclusivity (or allowed overlap) of the three health insurance phases, especially in two-member households with staggered retirement dates. |

## Passed Checks

- **Stream-based architecture consistency:** All time-varying quantities in the requirements reference stream paths from the conceptual model. No requirement introduces a non-stream time-varying value.
- **Denomination awareness in bedrock:** The four-frame denomination system (YZV/CYV/FYV/PYV) is well-defined in the bedrock principles. The requirement in [denomination.md](../requirements/engine/denomination.md) correctly references it.
- **Balance recurrence formula definitions:** The three formula variants (end-of-year, beginning-of-year, mid-year) in the conceptual model are complete and mathematically unambiguous. The MVP correctly selects end-of-year.
- **Withdrawal sequence requirement:** The withdrawal priority sequence requirement in [withdrawals.md](../requirements/engine/withdrawals.md) is specific and testable (bank first, then 401k, then Roth IRA).
- **Inflation differentiation:** The requirements correctly distinguish CPI (general), CPI-W (Social Security COLA), and health inflation, and correctly apply health inflation to all health expense streams.
- **Per-year override and carry-forward:** The assumption override pattern (per-year with carry-forward) is consistently applied across all assumption streams.
- **Retirement-year phase boundary:** The retirement year correctly serves as the phase boundary between working and retired phases, and expense streams correctly key off this boundary.

## Out of Scope

- **UX requirements:** Not reviewed. UI rendering, user input validation, and display formatting are outside engine QA scope.
- **Persistence, validation, stress-testing:** These files are placeholders and were not reviewed per instructions.
- **Design and implementation:** No design or implementation artifacts were reviewed. Findings are limited to whether requirements are testable and complete at the requirements level.
- **Controller requirements:** The controller mediates between engine and UX; its requirements were not reviewed.
