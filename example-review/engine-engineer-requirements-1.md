# Review: Engine Engineer
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The engine requirements provide a solid structural foundation rooted in the stream-based conceptual model. The core computation pipeline is identifiable: income, contributions, growth, withdrawals, expenses, and tax. However, several critical gaps exist: the requirements lack acceptance criteria entirely (violating requirements principles), multiple computational interactions are underspecified (surplus/deficit routing, tax feedback loops, Roth conversion mechanics), and the denomination requirements do not specify how the engine performs conversions. The requirements are sufficient to understand intent but insufficient to implement an unambiguous projection engine without significant design-time interpretation.

| <a id="EE-no-ac-CRT-01"></a>EE-no-ac-CRT-01 | All engine requirement files | No requirement in any engine file includes an Acceptance Criteria column. [Requirements principles](../requirements/principles.md) state: "A requirement without acceptance criteria is a wish. Every requirement must include conditions under which it can be declared satisfied." The specification table format defined in principles requires four columns: Path, Tag, Requirement, and Acceptance Criteria. Every engine file uses only three columns (Path, Tag, Requirement). | Add an Acceptance Criteria column to every engine requirements table and populate each row with observable, testable conditions. |
| <a id="EE-surplus-routing-CRT-02"></a>EE-surplus-routing-CRT-02 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / surplus-deficit` | The requirement states each year must produce "an observable surplus or deficit equal to total income after tax minus total expenses" but does not specify what happens with the surplus or deficit. A surplus must be routed somewhere (bank account? brokerage? contribution top-up?) and a deficit must be funded from somewhere (withdrawal sequence?). Without this, the balance recurrence has no net-flow input and the projection cannot close. | Add requirements specifying: (a) how surpluses are allocated across accounts, (b) how deficits trigger withdrawals via the withdrawal priority sequence, and (c) the relationship between surplus/deficit and the net-flow term in the balance recurrence formula. |
| <a id="EE-denom-convert-CRT-03"></a>EE-denom-convert-CRT-03 | [denomination.md](../requirements/engine/denomination.md) | The denomination file contains only two requirements: that every dollar amount must be denominated, and that display defaults to CYV. The bedrock principles define explicit conversion paths (YZV to CNV, YZV to YNV, etc.) but no engine requirement mandates that these conversions exist or specifies which conversions the engine must support. | Add requirements for each denomination conversion the engine must perform (minimally YZV to FYV for projection output, YZV to CYV for display, PYV to YZV for historical import). |
| <a id="EE-tax-circular-CRT-04"></a>EE-tax-circular-CRT-04 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / income-tax` and [withdrawals.md](../requirements/engine/withdrawals.md): `R:engine / withdrawals : sequence` | Tax computation depends on total income (which includes taxable withdrawals). The requirements do not specify an ordering or resolution strategy for this feedback loop. | Add a requirement specifying the computation ordering or iteration strategy that resolves the tax-withdrawal circularity (e.g., compute tax on known income first, then determine deficit, then add withdrawal income, then recompute tax; or specify that tax on withdrawals is computed in the same year vs. next year). |

| <a id="EE-net-flow-MAJ-01"></a>EE-net-flow-MAJ-01 | [projection.md](../requirements/engine/projection.md): `R:engine / projection / account-balance / recurrence : end-of-year` | The requirement states balances use `D:formulas / balance-recurrence / end-of-year` which is defined as `p(y) = p(y-1) * (1 + rate) + net-flow`. For account types that receive surplus routing or deficit-driven withdrawals, the composition of net-flow is even less clear. | Add a requirement that explicitly defines net-flow for each account type as the sum of all inflows (contributions, surplus allocation, conversions-in) minus all outflows (withdrawals, RMDs, conversions-out, fees) for that account in that year. |

## Passed Checks

- **Stream-based architecture consistency:** All time-varying quantities are modeled as streams consistent with bedrock principles. No requirement violates the stream abstraction.
- **Conceptual model linkage:** Every engine requirement correctly references its corresponding conceptual model entity using proper path syntax with markdown links.
- **Path syntax correctness:** All requirement paths follow the `R:` prefix convention and use correct `/`, `[ ]`, and `:` operators per the requirements principles.
- **Tag usage:** CORE and MVP tags are applied consistently. CORE is reserved for foundational invariants (denomination, timeline, account balance floor). MVP is used for first-release scope.
- **FUT tagging for alternate recurrence formulas:** Beginning-of-year and mid-year balance recurrence are correctly tagged as FUT.
- **Balance floor invariant:** The zero-floor constraint on account balances is correctly tagged CORE and applies to all accounts.
- **Health insurance phase sequencing:** The three insurance phases (employer, ACA, Medicare) correctly model the lifecycle transition, despite the multi-member ambiguity noted in findings.
- **Inflation differentiation:** The requirements correctly separate CPI (general), CPI-W (SS COLA), and health inflation, with appropriate defaults.
- **RMD regulatory parameters:** The assumptions correctly reference both the uniform lifetime table and joint survivor table, birth-year-to-start-age mapping, and penalty rates per SECURE 2.0.
- **Carry-forward semantics:** The requirement that per-year assumption overrides use carry-forward semantics is correctly specified and covers the stream-based override model.

## Out of Scope

- **UX requirements** were not reviewed per Leaf altitude scope restrictions.
- **Persistence, validation, and stress-testing files** are acknowledged as non-MVP placeholders and are not flagged.
- **Whether features are valuable** is a product decision, not an engine engineering concern. The missing-scope findings (HSA, brokerage, traditional IRA, Roth 401k) are flagged for completeness and implementability, not feature prioritization.
- **Design-level concerns** such as data structures, algorithm choice, and module boundaries are deferred to the design phase review.
