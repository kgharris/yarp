# Review: Database Engineer
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The requirements artifacts define a rich domain model anchored in streams, denominations, and configurable assumptions, but the MVP scope deliberately defers all persistence, validation, import/export, and schema evolution requirements. This creates a significant gap: the engine requirements implicitly depend on data durability (shipped defaults, per-year overrides, carry-forward semantics, policy tables, historical actuals) without specifying how that data enters the system, what guarantees protect it, or how it survives between sessions. Several data-integrity-relevant constraints are embedded in engine requirements (allocation sum check, denomination invariants, floor constraints) but have no corresponding persistence or input-validation requirements to enforce them at the data boundary.

## Findings

| ID | Location | Issue | Recommendation |
|---|---|---|---|
| <a id="DBE-persist-defer-CRT-01"></a>DBE-persist-defer-CRT-01 | [persistence.md](../requirements/engine/persistence.md) | Persistence is entirely deferred ("THE CONTENT THAT BELONGS IN THIS FILE IS NOT PART OF MVP"), yet the MVP engine requirements depend on persistent state. [assumptions.md](../requirements/engine/assumptions.md) requires "shipped default value sufficient to produce a complete projection without user configuration" (`R:engine / assumptions : defaults`), per-year overrides (`R:engine / assumptions : override`), and carry-forward semantics (`R:engine / assumptions : carry-forward`). These all presuppose stored configuration. Without any persistence requirement, the MVP has no defined mechanism for supplying, storing, or recovering plan inputs between sessions. | Add at minimum a narrow MVP persistence requirement covering: (a) the storage denomination for persisted dollar amounts (bedrock mandates YZV), (b) the round-trip guarantee that saved plan data reloads identically, and (c) a statement that the plan input file is the authoritative source of truth. Even a single-file "plan file" spec would close this gap. |
| <a id="DBE-valid-defer-CRT-02"></a>DBE-valid-defer-CRT-02 | [validation.md](../requirements/engine/validation.md) | Validation is entirely deferred, yet the MVP already contains implicit validation constraints: allocation sum must equal 100% (`R:engine / assumptions / allocations : sum`), SS claiming age must fall within FRA range (`R:engine / timeline / [ member ] : ss-claiming-age / range-constraint`), account balances must not go below zero (`R:engine / projection / account-balance : floor`), and the CLI must refuse to run if required assumptions are missing or allocations are invalid (`R:ux / cli / projection-table : missing-assumptions`, `R:ux / cli / projection-table : allocation-sum`). These are validation rules that must be enforced at data ingestion, not just at projection time. | Define an MVP-scoped subset of validation requirements covering: input validation at plan-load time (type correctness, range constraints, mandatory fields), and the error reporting contract when validation fails. This can be a small section within the deferred validation.md rather than a full validation framework. |
| <a id="DBE-data-mgmt-defer-CRT-03"></a>DBE-data-mgmt-defer-CRT-03 | [data-management.md](../requirements/ux/data-management.md) | Data management is entirely deferred ("NOT PART OF MVP"), but the CLI requirements assume a plan exists to project against. There is no requirement anywhere that describes how a plan is created, how the user provides initial inputs, or what the input mechanism is. The CLI spec (`R:ux / cli / projection-table`) requires producing a projection table but does not reference any input source. | Add a minimal MVP requirement for plan input specification: what format the user provides data in (e.g., a plan file path as CLI argument), what happens when the file is missing or malformed, and what constitutes a minimally valid plan. |
| <a id="DBE-storage-denom-MAJ-01"></a>DBE-storage-denom-MAJ-01 | [denomination.md](../requirements/engine/denomination.md), [bedrock-principles.md](../bedrock-principles.md) | Bedrock principles state "Storage is always YZV" and "All persisted configuration and assumption values are stored in year-zero dollars." However, the denomination requirements only specify two rows: the CORE invariant that every dollar amount has a denomination, and the MVP display default of CYV. There is no requirement stating the storage denomination. Since the conceptual model defines PYV as "always observed, never derived" and the assumptions include policy tables with year-specific dollar thresholds (bracket amounts, contribution limits, deduction amounts), the requirements must clarify which denomination applies to each category of stored data. | Add requirements specifying: (a) user-configured dollar amounts are stored in YZV, (b) policy/regulatory tables with year-specific published values (contribution limits, bracket thresholds) are stored in PYV for their reference year, (c) conversion between storage denomination and display denomination is explicit. |
| <a id="DBE-policy-data-MAJ-02"></a>DBE-policy-data-MAJ-02 | [assumptions.md](../requirements/engine/assumptions.md) | The requirements reference numerous policy data tables (401k limits, IRA limits, RMD tables, SS FRA tables, tax brackets, standard deduction, Medicare eligibility age, Roth IRA phase-out thresholds) with "shipped default" and "per-year override" semantics, but there is no requirement addressing how this policy data is sourced, versioned, or updated. The design principle "Data-Driven Business Rules" in [design/principles.md](../design/principles.md) mandates that these are data, not logic. The requirements should acknowledge that this data has a lifecycle: it is published externally (IRS, SSA), changes annually, and the user needs to know what vintage of policy data they are running against. | Add a requirement that policy data tables carry a version or effective-year indicator, and that the system must make the policy data vintage observable to the user (even if just via a version stamp in the plan file). |
| <a id="DBE-defaults-source-MAJ-03"></a>DBE-defaults-source-MAJ-03 | [assumptions.md](../requirements/engine/assumptions.md): `R:engine / assumptions : defaults` | "Every assumptions stream must have a shipped default value sufficient to produce a complete projection without user configuration." This requirement does not specify where defaults come from, whether they are embedded in the application binary or loaded from an external file, or whether the user can distinguish a default value from one they explicitly set. If defaults are baked into the binary, updating policy data requires a software release. If loaded from a file, the file becomes part of the data layer contract. | Clarify whether shipped defaults are considered application configuration or user data. Add a requirement that the system must distinguish between default-sourced and user-overridden values, so that when policy data is updated, user overrides are preserved while defaults refresh. |
| <a id="DBE-carry-forward-MAJ-04"></a>DBE-carry-forward-MAJ-04 | [assumptions.md](../requirements/engine/assumptions.md): `R:engine / assumptions : carry-forward` | "Per-year overrides must use carry-forward semantics." This is a data-layer-critical requirement because it defines how sparse data is interpreted. However, it lacks acceptance criteria. What happens at the boundary before the first override? Does the default value serve as the initial carry-forward anchor? What happens if a user removes an override for year Y -- does the value revert to the prior year's override or to the default? | Add acceptance criteria that specify: (a) carry-forward begins from the shipped default when no earlier override exists, (b) removing an override reverts to carry-forward from the nearest prior override (or default if none), (c) the behavior is deterministic for any combination of override insertions and deletions. |
| <a id="DBE-no-acceptance-MAJ-05"></a>DBE-no-acceptance-MAJ-05 | All engine requirement files | The requirements principles document states "A requirement without acceptance criteria is a wish. Every requirement must include conditions under which it can be declared satisfied." However, none of the engine requirement tables include an Acceptance Criteria column. The table schema is `Path | Tag | Requirement` with no fourth column. This is a systemic violation of [requirements/principles.md](../requirements/principles.md). From the data layer perspective, this is particularly concerning for constraints that affect data integrity (allocation sum, balance floor, denomination invariant) where the acceptance criteria would define the boundary behavior. | Add an Acceptance Criteria column to every engine requirements table, or document why the current requirement statements are considered self-evident. At minimum, data-integrity constraints need explicit acceptance criteria defining edge-case behavior. |
| <a id="DBE-denom-naming-MIN-01"></a>DBE-denom-naming-MIN-01 | [bedrock-principles.md](../bedrock-principles.md) vs [conceptual-model.md](../requirements/conceptual-model.md) | Bedrock principles use abbreviations CNV, YNV, PNV for denomination frames. The conceptual model uses CYV, FYV, PYV. These refer to the same concepts but use different abbreviations. `CNV` (Current Nominal Value) vs `CYV` (Current Year Value); `YNV` (Year Nominal Value) vs `FYV` (Future Year Value); `PNV` (Past Nominal Value) vs `PYV` (Past Year Value). Only YZV is consistent across both documents. This inconsistency is a data integrity risk: if schema fields, API contracts, or serialized files use one naming convention while code uses another, silent misinterpretation is possible. | Reconcile the abbreviations to a single set across bedrock principles and the conceptual model. The conceptual model's CYV/FYV/PYV naming is arguably clearer (year-centric rather than "nominal"), but the choice should be explicit and universal. |
| <a id="DBE-member-identity-MIN-02"></a>DBE-member-identity-MIN-02 | [household.md](../requirements/engine/household.md), [conceptual-model.md](../requirements/conceptual-model.md) | The conceptual model defines `S:household / [ member ]` as a set node, meaning each member is an instance. But there is no requirement specifying how members are identified -- by name, by ordinal position, by a stable identifier? Every stream that references `[ member ]` (accounts, contributions, withdrawals, income, expenses) depends on a stable member identity to associate data correctly. If member identity is positional (first vs second), renaming or reordering could corrupt cross-references. | Add a requirement that each household member has a stable, unique identifier within the plan, and that all member-scoped streams reference members by this identifier. |
| <a id="DBE-account-identity-MIN-03"></a>DBE-account-identity-MIN-03 | [assets.md](../requirements/engine/assets.md), [conceptual-model.md](../requirements/conceptual-model.md) | Similar to member identity, account instances within set nodes (`[ 401k ]`, `[ roth-ira ]`, `[ bank ]`) have no specified identity mechanism. The conceptual model allows multiple accounts of the same type per member (e.g., multiple 401k accounts from different employers). Without a stable identity scheme, persisted references from contributions, withdrawals, and allocation overrides to specific accounts become fragile. | Add a requirement that each account instance has a stable, unique identifier within the plan, and that all account-scoped streams reference accounts by this identifier. |
| <a id="DBE-birth-year-MIN-04"></a>DBE-birth-year-MIN-04 | [timeline.md](../requirements/engine/timeline.md), [household.md](../requirements/engine/household.md) | `R:engine / timeline / [ member ]` requires "a derivable age for every year in the projection range," which implies a birth year is stored per member. However, no requirement explicitly states that birth year is a required input for each member. Birth year is critical for RMD start age derivation (`R:engine / timeline / [ member ] : rmd-start-age`), SS FRA derivation (`R:engine / assumptions / policy / ss / fra`), and age-based eligibility thresholds. It is a foundational data field that should be explicitly required. | Add an explicit requirement that each household member must have a birth year, and that it is a mandatory input. |
| <a id="DBE-json-output-MIN-05"></a>DBE-json-output-MIN-05 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / output-format : json` | "The CLI must support JSON-formatted output for use by test infrastructure." This establishes JSON as a data contract but does not specify whether the JSON schema is stable, versioned, or documented. If test infrastructure depends on it, changes to the output structure would break tests silently. From a data layer perspective, any machine-readable output format is a schema commitment. | Add a requirement that the JSON output schema is versioned or documented sufficiently that consumers can detect breaking changes. Alternatively, state that the JSON output format is internal to the test infrastructure and not a public contract. |
| <a id="DBE-anchor-year-QST-01"></a>DBE-anchor-year-QST-01 | [bedrock-principles.md](../bedrock-principles.md), [denomination.md](../requirements/engine/denomination.md) | Bedrock principles state "YZV is fixed at the plan anchor year and does not change as the calendar advances (unless the user explicitly rebases)." This implies a rebase operation exists. However, there is no requirement for rebase anywhere in the requirements tree, and no mention of what happens to stored YZV values when a rebase occurs. Rebase is a data migration operation with integrity implications: every stored dollar amount would need conversion. | Clarify whether rebase is in scope (even as FUT) and, if so, add it to the requirements roadmap. If not in scope, consider adding a note that the anchor year is immutable once set, to prevent confusion. |
| <a id="DBE-plan-creation-QST-02"></a>DBE-plan-creation-QST-02 | [timeline.md](../requirements/engine/timeline.md): `R:engine / timeline` | The timeline is CORE and "must span a defined range of projection years per D:streams / timeline." The conceptual model says "The plan timeline spans from the earliest stream start to the latest stream end. It is not independently configured." This means the timeline is derived from the data. But what is the expected behavior when a plan has no streams yet (initial creation)? Is an empty plan valid? Can a projection run on an empty plan? | Clarify the minimal data set required for a valid plan. Define what constitutes a "minimally projectable" plan -- likely at least one member with a birth year and at least one income or account stream. |
| <a id="DBE-override-denom-QST-03"></a>DBE-override-denom-QST-03 | [assumptions.md](../requirements/engine/assumptions.md): `R:engine / assumptions : override` | "Any assumptions stream must be overridable on a per-year basis." For dollar-denominated assumptions (tax bracket thresholds, contribution limits, standard deduction), what denomination does the user provide the override in? If the user overrides the 2045 standard deduction, do they enter it in YZV, CYV, or FYV for 2045? Bedrock says storage is YZV, but requiring the user to think in year-zero dollars may be unintuitive. The conversion between user-entry denomination and storage denomination needs to be specified. | Add a requirement (or reference to the denomination requirements) specifying the denomination in which per-year overrides are entered and how they are converted to storage denomination. |

## Passed Checks

- **Denomination invariant** (`R:engine / denomination`): The CORE requirement that every dollar amount has exactly one denomination is clearly stated and aligns with bedrock principles.
- **Stream model consistency**: All engine requirements correctly reference conceptual model stream paths using the established path syntax. Cross-references between requirements files (e.g., contributions referencing assets, withdrawals referencing assumptions) are internally consistent.
- **Tag usage**: CORE, MVP, and FUT tags are applied consistently across all requirements files. CORE is reserved for true invariants (denomination, timeline span, balance floor). FUT is used appropriately for deferred capabilities (beginning-of-year and mid-year recurrence formulas).
- **Separation of concerns**: Engine requirements do not prescribe data formats, field names, or storage mechanisms. The requirements remain technology-agnostic as mandated by [requirements/principles.md](../requirements/principles.md).
- **Conceptual model coverage**: The conceptual model provides a comprehensive domain entity catalog. Every stream referenced in engine requirements traces back to a defined path in the conceptual model.
- **Expense phasing**: Health insurance expense phase transitions (employer -> ACA -> Medicare) are well-defined with clear boundary conditions tied to member age and retirement year.

## Out of Scope

- **Financial rule correctness**: Whether the RMD computation formula, SS COLA adjustments, or tax bracket application logic is financially accurate is outside this review's domain. Deferred to the Financial Domain Reviewer and Engine Engineer.
- **UX layout and interaction design**: The deferred UX requirements files (dashboard, projections-view, budget-view, charts, scenarios, estate, balance-sheets) were not reviewed. Only [cli.md](../requirements/ux/cli.md) and [data-management.md](../requirements/ux/data-management.md) were surveyed for data contracts.
- **Projection formula correctness**: Whether end-of-year recurrence is the right default, or whether the weighted-average rate derivation is mathematically sound, is outside this review's domain.
- **Code style and implementation specifics**: Not applicable at the requirements phase.
