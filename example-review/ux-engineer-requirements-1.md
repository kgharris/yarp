# Review: UX Engineer
**Phase:** requirements | **Iteration:** 1 | **Date:** 2026-04-08

## Summary

The only MVP-scoped UX artifact is [cli.md](../requirements/ux/cli.md), which defines six requirements for command-line projection output. The requirements capture the core intent -- produce a year-by-year projection table with denomination-aware monetary values -- but are structurally incomplete: the table is missing the mandatory Acceptance Criteria column, several columns reference domain concepts without enough specificity to build from, and error/validation requirements describe what must not happen without specifying what the user observes instead. The remaining eight UX files are intentionally deferred placeholders and were not flagged.

## Findings

| ID | Location | Issue | Recommendation |
|---|---|---|---|
| <a id="UXE-no-ac-CRT-01"></a>UXE-no-ac-CRT-01 | [cli.md](../requirements/ux/cli.md): all rows | The requirements table has three columns (Path, Tag, Requirement) but [requirements principles](../requirements/principles.md) mandate four: Path, Tag, Requirement, **Acceptance Criteria**. Every requirement in this file lacks acceptance criteria. Per principles: "A requirement without acceptance criteria is a wish." | Add an Acceptance Criteria column to the table and populate every row with observable conditions under which the requirement is satisfied. |
| <a id="UXE-no-error-feedback-CRT-02"></a>UXE-no-error-feedback-CRT-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : missing-assumptions`, `R:ux / cli / projection-table : allocation-sum` | Both validation requirements state the projection "must not run" but specify no user-observable behavior when validation fails. The user has no way to know what went wrong. A CLI that silently refuses to run or crashes with an unspecified error is unbuildable as a UX requirement. | Add requirements specifying that validation failures produce a clear, actionable error message identifying which assumptions are missing or which allocations fail the sum check. |
| <a id="UXE-surplus-deficit-MAJ-01"></a>UXE-surplus-deficit-MAJ-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The column list includes "surplus or deficit" without defining what this value represents. Is it total income minus total expenses? Does it include contributions and withdrawals? Does it account for tax? The conceptual model has no `S:surplus-deficit` stream or definition to reference. A front-end engineer cannot build this column without knowing its formula inputs. | Either define surplus/deficit in the [conceptual model](../requirements/conceptual-model.md) or add a requirement that specifies its composition in terms of existing streams (income, expenses, contributions, withdrawals, tax). |
| <a id="UXE-fed-tax-ambiguous-MAJ-02"></a>UXE-fed-tax-ambiguous-MAJ-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | "Federal tax" appears in the column list without specifying what it represents: total federal income tax liability, effective rate, marginal rate, or some composite including FICA/SE/NIIT. The conceptual model defines multiple tax components (`S:assumptions / policy / tax / federal / brackets`, `S:assumptions / policy / tax / niit`, `S:assumptions / policy / tax / se`) but the column requirement does not indicate which are included or how they are aggregated. | Clarify whether "federal tax" means total federal tax liability (income + FICA + NIIT), or just income tax, and whether it is a single aggregate number or broken into components. Reference the relevant conceptual model paths. |
| <a id="UXE-no-contribs-withdrawals-MAJ-03"></a>UXE-no-contribs-withdrawals-MAJ-03 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The column list includes account balances, income, expenses, federal tax, and surplus/deficit, but omits [S:contributions](../requirements/conceptual-model.md#s-contributions) and [S:withdrawals](../requirements/conceptual-model.md#s-withdrawals). These are core streams in the conceptual model and are essential for understanding how balances change year over year. Without them the projection table cannot tell the user why a balance went up or down. | Either add contributions and withdrawals to the required columns, or document why they are intentionally excluded from the MVP projection table. |
| <a id="UXE-income-by-source-MAJ-04"></a>UXE-income-by-source-MAJ-04 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | "Income by source" is ambiguous. The conceptual model defines income at multiple levels: by type (`S:income / w2`, `S:income / ss`, `S:income / pension`, etc.) and by instance within type (`S:income / w2 / [ member ] / [ employer ]`). Does "by source" mean one column per income type, one column per income stream instance, or one column per member? This affects column count and readability significantly in a CLI table. | Specify the grouping level for income columns. For a CLI table, a per-type or per-member-per-type grouping is likely most practical, but this should be stated explicitly. |
| <a id="UXE-no-input-spec-MAJ-05"></a>UXE-no-input-spec-MAJ-05 | [cli.md](../requirements/ux/cli.md): general | The file specifies output requirements but contains no requirements for CLI input. How does the user provide plan data to the CLI? Is it a file path argument, stdin, a default location? The validation requirements (`missing-assumptions`, `allocation-sum`) imply input is loaded before projection, but the input mechanism is unspecified. | Add requirements for how the CLI receives plan data -- file path argument, default file location, or other mechanism -- and what happens when the input file is missing or malformed. |
| <a id="UXE-json-unspecified-MAJ-06"></a>UXE-json-unspecified-MAJ-06 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / output-format : json` | "The CLI must support JSON-formatted output for use by test infrastructure" specifies a format but not its content or structure. Does JSON output contain the same columns as the table? Is it the full projection data including intermediate values? Is it a flat array of year objects or a nested structure? Without content requirements, this is unbuildable. | Specify whether the JSON output mirrors the projection table columns or exposes a richer data set, and add acceptance criteria for what a valid JSON output contains. |
| <a id="UXE-state-tax-absent-MIN-01"></a>UXE-state-tax-absent-MIN-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The column list includes "federal tax" but not state tax, despite the conceptual model defining `S:assumptions / policy / tax / state / rate`. If state tax is computed by the engine, its absence from the projection table is a gap. If it is intentionally excluded from MVP, this should be noted. | Clarify whether state tax is included in the projection table. If excluded from MVP, tag it as FUT or note the exclusion explicitly. |
| <a id="UXE-denom-mode-MIN-02"></a>UXE-denom-mode-MIN-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : denomination` | The requirement says monetary values must be displayed per `D:denomination / display`, which the conceptual model defines as defaulting to CYV. However, the bedrock principles note that users may switch to year-zero display mode, and the denomination model includes FYV for projection output. There is no CLI requirement for the user to control which denomination mode is used. | Add a requirement (MVP or FUT) for a CLI flag or argument that allows the user to select denomination mode (CYV, YZV, FYV). If deferred, tag it explicitly. |
| <a id="UXE-liabilities-absent-MIN-03"></a>UXE-liabilities-absent-MIN-03 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The column list omits [S:liabilities](../requirements/conceptual-model.md#s-liabilities) entirely. Amortized loans, interest-only loans, and credit lines are defined in the conceptual model and affect both cash flow and net worth, yet they do not appear in the projection table. | Either add liability balances and/or payments to the required columns, or document why they are excluded from the MVP projection table. |
| <a id="UXE-wide-table-MIN-04"></a>UXE-wide-table-MIN-04 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : columns` | The required columns include ages for all household members, balances for all accounts, and income by source. For a two-member household with even a modest number of accounts and income sources, this will produce a very wide table that may be unreadable in a standard terminal. There is no requirement addressing table readability, column truncation, or overflow behavior. | Add a requirement or design consideration for how the CLI handles wide tables -- wrapping, pagination, column filtering, or other readability strategy. |
| <a id="UXE-allocation-scope-QST-01"></a>UXE-allocation-scope-QST-01 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : allocation-sum` | The requirement says projection must not run if allocations do not sum to 100%. The conceptual model defines allocations at both the global level (`S:assumptions / allocations`) and per-account level (`S:assumptions / allocations / [ member ] / [ account ]`). Does the 100% sum check apply to the global allocation, each per-account override, or both? | Clarify scope of the allocation sum validation -- global only, per-account only, or all allocation sets. |
| <a id="UXE-missing-assumptions-scope-QST-02"></a>UXE-missing-assumptions-scope-QST-02 | [cli.md](../requirements/ux/cli.md): `R:ux / cli / projection-table : missing-assumptions` | "Required assumptions" is not defined. Which assumptions are required vs. optional? The conceptual model lists assumptions for inflation, rates, allocations, and extensive policy parameters. If a user provides inflation and rates but no policy data, does the projection refuse to run? This needs a defined boundary or a reference to where "required" is specified. | Define the set of required assumptions explicitly, or reference an engine requirement that enumerates them. |

## Passed Checks

- **Conceptual model references are well-formed:** All links in cli.md use markdown link syntax and point to valid anchors in the conceptual model, consistent with the markdown formatting rules.
- **Path syntax is correct:** All paths use the `R:` prefix for requirement paths, `D:` for definition references, and `S:` for stream references, consistent with requirements principles.
- **Tag usage is correct:** All requirements are tagged `MVP`, which is a valid tag per requirements principles.
- **No solutioning detected:** Requirements describe observable behavior (table output, validation refusal, JSON format) without prescribing UI components, data structures, or algorithms.
- **One behavior per requirement:** Each row addresses a single concern -- no compound requirements detected.
- **Domain language used throughout:** Requirements use financial planning vocabulary, not technology jargon.
- **Placeholder files are properly marked:** All eight non-MVP UX files clearly state they are deferred and list their future scope.

## Out of Scope

- **Engine requirements:** Not reviewed. The column specification in cli.md references engine output (balances, income, expenses, tax), but correctness of engine computation is not in UX scope.
- **Data persistence requirements:** Not reviewed. How plan data is stored and loaded is engine/DB scope, though the absence of CLI input specification (UXE-no-input-spec-MAJ-05) touches this boundary.
- **Design-level concerns:** Column layout, terminal rendering strategy, and JSON schema structure are design concerns -- findings here are limited to whether requirements are specific enough to drive design.
